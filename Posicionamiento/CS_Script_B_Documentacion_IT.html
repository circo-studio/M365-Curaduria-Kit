01 Prerequisitos Requiere acción de IT
Escenario A · Usuario del cliente

El usuario que va a ejecutar el script necesita cumplir dos condiciones antes de correrlo:

01
Registrar la App Registration en Entra ID del propio tenant
Se crea una aplicación en Entra ID con permisos delegados — es decir, la app actúa en nombre del usuario que se autentica, con el alcance de permisos de ese usuario. Si el usuario no tiene acceso a un sitio, el script tampoco lo tendrá.
Entra ID → App registrations → New registration
Nombre sugerido: CuraduriaM365-ReadOnly
Tipo de cuenta: Single tenant
Redirect URI: https://login.microsoftonline.com/common/oauth2/nativeclient
Tipo de permisos: Delegated (en nombre del usuario)
02
El usuario debe tener acceso a los sitios a relevar
Con permisos delegados, el script hereda el nivel de acceso del usuario autenticado. Para leer todas las bibliotecas de un sitio sin restricciones, el usuario necesita ser Site Collection Administrator o tener acceso de lectura total en cada sitio GR- marcado para relevar.
Opción 1: Asignar rol SharePoint Administrator al usuario (acceso a todos los sitios)
Opción 2: Asignar SCA en cada sitio GR- individualmente (más granular)
Opción 3: Usar Script A AsignarOwner para asignar el SCA en lote sobre los sitios del inventario
02 App Registration — permisos delegados Entra ID
Escenario A · Usuario del cliente

Con permisos delegados, la App Registration es el vehículo de autenticación pero el alcance real lo define el usuario. La app no tiene permisos propios — toma los del usuario que se loguea.

Conexión — autenticación interactiva delegada
# El usuario del cliente se autentica con su cuenta corporativa + MFA # -Interactive abre el browser; -PersistLogin cachea el token para el loop Connect-PnPOnline -Url "https://[tenant].sharepoint.com" -Interactive -ClientId "[ClientId]" -PersistLogin # El token obtenido refleja los permisos del usuario autenticado # Si el usuario no tiene acceso a un sitio → el script reporta ErrorConexion para ese sitio
Ventaja de este escenario

No hace falta invitar ninguna cuenta externa al tenant. No hay cuenta B2B guest. Todo el acceso está acotado por los permisos del usuario que ejecuta — lo que simplifica la gestión y la auditoría desde IT.

01 Prerequisitos Requiere acción de IT
Escenario B · Circo Studio

Antes de que Circo Studio pueda ejecutar el script, IT del cliente necesita completar dos acciones en el tenant:

01
Registrar la App Registration en Entra ID del cliente
Se crea una aplicación en Entra ID con permisos de aplicación — no delegados. La app actúa con sus propios permisos, independientemente de qué usuario la invoca. Circo Studio provee el Client ID que va en la configuración del script.
Entra ID → App registrations → New registration
Nombre sugerido: CuraduriaM365-Circo-ReadOnly
Tipo de cuenta: Single tenant
Redirect URI: https://login.microsoftonline.com/common/oauth2/nativeclient
Tipo de permisos: Application (no en nombre de usuario)
02
Invitar la cuenta B2B de Circo Studio y asignarla como SCA
La cuenta guest de Circo Studio se invita al tenant y se asigna como Site Collection Administrator en los sitios GR- a relevar. Este acceso es temporal y se revoca al finalizar el engagement. El Script A AsignarOwner automatiza esta asignación en lote.
Alcance: Solo sitios GR- Legacy marcados para relevar
Rol: Site Collection Administrator (temporal)
Revocación: Notificación formal de Circo Studio al cierre del engagement
02 App Registration — permisos de aplicación Entra ID
Escenario B · Circo Studio

Con permisos de aplicación, la App Registration tiene sus propios permisos independientes de cualquier usuario. La sesión siempre requiere MFA interactivo — no existe ningún secreto almacenado que permita conexión automática.

Conexión — autenticación interactiva con cuenta B2B
# La cuenta B2B de Circo Studio completa MFA en el browser del operador # -ForceAuthentication evita reutilizar tokens de sesiones anteriores # -PersistLogin cachea el token para reutilizar en el loop sin repetir MFA Connect-PnPOnline -Url "https://[tenant].sharepoint.com" -Interactive -ClientId "[ClientId]" -ForceAuthentication # Cada sitio del loop reutiliza el token sin nuevo login: Connect-PnPOnline -Url $sitioUrl -ClientId $clientId -PersistLogin
Por qué Interactive y no un secreto de aplicación

El script usa autenticación interactiva en lugar de un secreto o certificado. Esto significa que la sesión siempre requiere la presencia física de un operador de Circo Studio que complete el flujo de MFA. No existe ninguna credencial almacenada que pueda usarse sin intervención humana.

05 Flujo de ejecución Igual en ambos escenarios
01
Carga de configuración
Lee el cliente_config.json: URL del tenant, Client ID, carpeta de salida, lista de sitios ya analizados y prefijos de canales a excluir.
02
Lectura del CSV de inventario y filtrado de sitios
Carga el CSV del Script A Inventario y aplica los filtros: solo sitios GR- Legacy, excluye bloqueados (ReadOnly), excluye ya analizados por ID, excluye canales de padres cubiertos. Ordena por storage descendente — los sitios más grandes se procesan primero.
03
Login único con token cacheado
Esta es la única autenticación de toda la ejecución. El browser se abre para MFA. El token resultante se cachea con -PersistLogin y se reutiliza para conectar a cada sitio del loop sin volver a pedir credenciales.
04
Loop por sitio — checkpoint, conexión, extracción
Para cada sitio: verifica checkpoint (si ya existe CSV, lo saltea), conecta reutilizando el token, obtiene las bibliotecas de documentos, itera los archivos con paginado de 100 ítems y escribe por lotes al CSV de salida sin acumular en memoria.
05
Resumen parcial guardado después de cada sitio
Después de procesar cada sitio se escribe una fila al CSV de resumen. Si la ejecución se interrumpe en cualquier punto, el resumen parcial ya está guardado y los sitios con CSV existente no se vuelven a procesar.
06
Resumen final
Imprime y guarda: sitios completados, salteados, sin contenido y con error. Desconecta la sesión.
06 Paginado y gestión de memoria Sin timeout ni acumulación

El script resuelve dos problemas clásicos al iterar grandes volúmenes en SharePoint Online: timeout por operaciones largas y agotamiento de memoria por acumular resultados.

Flujo de cada página de 100 ítems
SPO · Biblioteca
→
Lote 100 ítems
ScriptBlock · en memoria
→
CSV · disco local
Export-Csv -Append
Cada lote se escribe inmediatamente y se descarta de memoria. En ningún momento se acumulan todos los archivos de un sitio en RAM.
Núcleo del mecanismo — ScriptBlock con escritura por lote
# PageSize 100: evita el List View Threshold de SPO (límite de 5.000 ítems por consulta) # -ScriptBlock ejecuta este bloque por cada página recibida, sin acumular Get-PnPListItem -List $bib.Title -PageSize 100 -Fields $Fields -ScriptBlock { Param($items) # Generic.List es O(1) por Add(); @() += es O(n²) — crítico en miles de archivos $lote = [System.Collections.Generic.List[PSObject]]::new() foreach ($item in $items) { if ($item.FileSystemObjectType -eq "File") { $lote.Add([PSCustomObject]@{ ... metadatos ... }) } } # Escribir al CSV inmediatamente — el primer lote crea el archivo con headers if ($script:primerLote) { $lote | Export-Csv -Path $csvSitio -NoTypeInformation -Encoding UTF8 $script:primerLote = $false } else { $lote | Export-Csv -Path $csvSitio -NoTypeInformation -Encoding UTF8 -Append } # $lote sale de scope → GC libera memoria antes del próximo lote }
Por qué PageSize 100 y no más grande

SharePoint Online tiene un límite de 5.000 ítems por consulta REST (List View Threshold). PageSize 100 divide la iteración en llamadas pequeñas que nunca alcanzan ese límite. Una biblioteca con 50.000 archivos se procesa con 500 llamadas de 100 ítems sin timeout y sin romper ningún límite de la plataforma.

07 Checkpoint e idempotencia Seguro para interrumpir

El script puede interrumpirse en cualquier momento y volver a ejecutarse sin reprocesar los sitios ya completados.

Lógica de checkpoint — al inicio de cada sitio
# Buscar si ya existe un CSV de salida para este sitio $existente = Get-ChildItem $carpetaSalida -Filter "*B_Docs_${nombreLimpio}_*.csv" | Select-Object -First 1 if ($existente) { # Ya fue relevado — saltear sin conectar al sitio return @{ Estado = "Salteado"; CsvPath = $existente.FullName } } # Si no existe CSV → proceder con el relevamiento completo
Sitio interrumpido a mitad

Si el script se corta mientras procesa un sitio, ese sitio queda con un CSV parcial. En la próxima ejecución el checkpoint lo detecta como "ya procesado" y lo saltea.

Para forzar el re-relevamiento de un sitio puntual: eliminar su CSV de la carpeta de salida y volver a ejecutar. El Script C Refresh también cubre esta situación.

11 Auditoría y trazabilidad Auditable por Microsoft
A
Log local del script
El archivo _LOG.txt registra cada conexión, biblioteca procesada, conteo de archivos por lote, errores y resumen final con timestamp por línea.
B
Unified Audit Log de Microsoft 365
Toda actividad de la App Registration y la cuenta que ejecuta queda registrada en el Unified Audit Log del tenant, consultable desde el Microsoft Purview Compliance Portal.
Purview → Audit → Search
Actividad: FileAccessed, FileAccessedExtended, ListItemsViewed
Usuario: cuenta del ejecutor o App Registration ID
C
Revocación de acceso al finalizar (Escenario B)
Al concluir el engagement, Circo Studio notifica formalmente al cliente para que revoque en tres puntos:
1. Remover la cuenta B2B del rol SCA en todos los sitios
2. Deshabilitar o eliminar la App Registration en Entra ID
3. Verificar en el Unified Audit Log que no hubo actividad posterior
Resumen de la postura de seguridad

El script es de solo lectura, requiere MFA interactivo en cada sesión, no almacena credenciales, no sube datos al tenant, no accede al contenido de archivos y toda su actividad es auditable desde Microsoft Purview.
