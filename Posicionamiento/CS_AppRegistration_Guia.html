Kit M365 Curaduria · Configuracion de acceso al tenant

App Registration en Entra ID
Guia de configuracion y alcance de seguridad

Todo acceso del kit al tenant del cliente pasa por una App Registration en Entra ID. Este documento explica que es, como crearla, que permisos asignar, que implica en terminos de seguridad y como configurarla en el kit.

Para quien
Consultor Circo Studio + IT del cliente
Portal
portal.azure.com / Entra ID
Tiempo estimado
15 a 20 minutos

El kit se conecta al tenant de SharePoint Online usando una App Registration — una identidad de aplicacion registrada en Entra ID (antes Azure Active Directory) del cliente. Este es el mecanismo oficial de Microsoft para que aplicaciones externas accedan a un tenant de forma controlada, con permisos explicitos y actividad completamente auditable.

La alternativa seria usar credenciales de usuario — usuario y contrasena de una cuenta del tenant. Eso tiene tres problemas graves: expone credenciales personales, no permite permisos granulares y no distingue en el audit log si fue una persona o un script quien accedio. La App Registration resuelve los tres.

Sin App Registration
Credenciales de usuario
Usuario y contrasena expuestos en el script
Acceso heredado del usuario — sin granularidad
El audit log muestra el usuario, no la aplicacion
Si el usuario cambia su contrasena, el script falla
No hay forma de revocar solo el acceso del script
Con App Registration
Identidad de aplicacion
Sin credenciales almacenadas — solo Client ID
Permisos explicitos — solo lo que se declara
Audit log muestra la app por nombre y ID
Independiente de cualquier usuario del tenant
Revocable en un click desde Entra ID
El Client ID no es una contrasena

El Client ID es el identificador publico de la app — equivalente al nombre de usuario pero sin la contrasena. Solo con el Client ID no se puede acceder a nada. El acceso real requiere ademas autenticacion interactiva con MFA por parte del operador que ejecuta el script. Por eso el kit no almacena secretos ni certificados — toda sesion requiere presencia humana.

01 Crear la App Registration Requiere Global Admin o Application Admin

La App Registration se crea una sola vez por tenant. La crea el administrador de IT del cliente, no Circo Studio.

01
Abrir el portal de Entra ID
portal.azure.com → Entra ID → App registrations
O ingresar directamente a entra.microsoft.com y buscar App registrations en el menu lateral.
02
Nueva registro
Hacer click en New registration (boton azul en la parte superior).
Formulario de registro
Name CuraduriaM365-Circo-ReadOnly
Supported account types Accounts in this organizational directory only (Single tenant)
Redirect URI (optional) Public client/native → https://login.microsoftonline.com/common/oauth2/nativeclient
El nombre es libre — elegir uno que identifique claramente el proposito. El Redirect URI es necesario para que PnP.PowerShell pueda completar el flujo de autenticacion interactiva.
03
Confirmar tipo de plataforma
Despues de crear el registro, ir a Authentication en el menu lateral y verificar que el Redirect URI este bajo Mobile and desktop applications (no bajo Web). Si no esta, agregarlo ahi manualmente.
Por que Mobile and desktop

PnP.PowerShell corre desde la linea de comandos — es una aplicacion de escritorio, no una app web. El tipo de plataforma determina el flujo de autenticacion OAuth. Con el tipo incorrecto, el login interactivo falla o no completa el MFA correctamente.

04
Habilitar Allow public client flows
En la misma pantalla de Authentication, al final de la pagina, activar la opcion Allow public client flows → Yes. Guardar.
Esto permite que PnP complete el flujo de autenticacion desde PowerShell sin necesitar un secreto de aplicacion almacenado.

PnP.PowerShell necesita este permiso para poder actuar en el contexto de un Site Collection Administrator — que es el rol minimo para acceder a todas las bibliotecas de un sitio, incluidas algunas que tienen permisos restringidos.

El permiso en si es amplio pero el uso real es limitado. El script solo ejecuta Get-PnPList y Get-PnPListItem — ambas operaciones de lectura. Ninguna llamada del kit crea, modifica ni elimina contenido.

Si la politica de seguridad del cliente no permite Sites.FullControl.All de forma permanente, puede otorgarse como permiso temporal y revocarse al finalizar el engagement.

03 Otorgar Admin Consent Requiere Global Admin

Los permisos de tipo Application (no delegados) requieren que un Global Administrator del tenant los apruebe explicitamente. Sin este paso, la App Registration existe pero no puede acceder a nada.

01
Otorgar consentimiento
App Registration → API permissions → Grant admin consent for [tenant]
El boton esta en la parte superior de la lista de permisos. Al hacer click, el Global Admin confirma que aprueba los permisos para toda la organizacion.
02
Verificar que el consent fue otorgado
Despues del consent, cada permiso debe mostrar un tilde verde en la columna Status con el texto Granted for [NombreTenant]. Si alguno muestra "Not granted", repetir el paso anterior.
Estado esperado en API permissions
Sites.Read.All✓ Granted for [Tenant]
Sites.FullControl.All✓ Granted for [Tenant]
User.Read.All✓ Granted for [Tenant]
04 Obtener el Client ID Lo que va en el kit

El Client ID es el identificador unico de la App Registration dentro del tenant. Es el unico valor que hay que pasar al kit — no hay contrasenas ni secretos.

01
Ir a la pantalla Overview de la App Registration
App Registration → Overview
02
Copiar el Application (client) ID
En la pantalla Overview aparecen tres IDs. El que se necesita es Application (client) ID — el primero de la lista. Tiene formato UUID.
Overview de la App Registration
Display nameCuraduriaM365-Circo-ReadOnly
Application (client) IDxxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx ← este
Object IDno usar este
Directory (tenant) IDno usar este
Hacer click en el icono de copiar al lado del Application (client) ID. Este UUID es lo que va en el campo Client ID del editor de configuracion del kit.
Client ID · formato esperado xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
No confundir con el Object ID ni el Tenant ID

Los tres IDs en la pantalla Overview tienen el mismo formato UUID. El kit usa exclusivamente el Application (client) ID. El Object ID identifica el objeto en el directorio y el Tenant ID identifica al tenant — ninguno de los dos sirve para autenticar la app.

05 Escenarios de ejecucion Dos modelos de acceso

El kit soporta dos escenarios de ejecucion con implicancias distintas en permisos y auditoria:

Escenario A
Un usuario del cliente ejecuta el script
La App Registration usa permisos delegados
El acceso esta acotado al nivel del usuario que se loguea
Solo necesita Sites.Read.All delegado
No se invita ninguna cuenta externa al tenant
El audit log muestra la cuenta del usuario + la app
Menor superficie de acceso
Escenario B
Circo Studio ejecuta el script
La App Registration usa permisos de aplicacion
Se invita la cuenta B2B de Circo Studio como guest
Requiere Sites.FullControl.All application
La cuenta B2B se asigna como SCA en sitios especificos
El audit log muestra la cuenta B2B + el Client ID de la app
Mayor trazabilidad de las acciones de Circo
Cual elegir

Si el cliente tiene un administrador de IT disponible para ejecutar los scripts, el Escenario A es preferible — menor privilegio, sin cuenta externa. El Escenario B es el flujo estandar cuando Circo Studio lleva el engagement de forma autonoma. En ambos casos la App Registration es la misma — solo cambian los tipos de permisos configurados.

06 Alcance real del acceso Solo lectura de metadatos

Tener un permiso en Entra ID no significa que el kit use todo el alcance de ese permiso. Los permisos son el techo — lo que el kit realmente hace es el piso. Esta tabla muestra la diferencia:

Permiso en Entra IDLo que permite tecnicamenteLo que el kit usa en la practica
Sites.Read.All
Leer cualquier contenido de cualquier sitio SPO del tenant
Leer metadatos de archivos (nombre, tamano, fechas, autor) en sitios GR- marcados para relevar
Sites.FullControl.All
Crear, modificar y eliminar contenido en cualquier sitio SPO
Conectar al sitio como SCA para poder leer todas las bibliotecas. Ninguna operacion de escritura.
User.Read.All
Leer el perfil completo de todos los usuarios del tenant
Resolver el nombre de display del autor de cada archivo (LookupValue del campo Author)
Lo que el kit nunca hace

El kit no descarga el contenido de ningun archivo. No modifica permisos. No crea ni elimina elementos. No accede a Teams, OneDrive personal, buzones ni conversaciones. No accede a sitios que no esten en la lista de sitios a relevar del config JSON.

07 Auditoria y control desde el tenant Auditable en Microsoft Purview

Toda la actividad de la App Registration queda registrada en el Unified Audit Log de Microsoft 365, accesible desde el Microsoft Purview Compliance Portal. IT puede consultar exactamente que accedio, cuando y desde que cuenta.

A
Consultar actividad en Purview
compliance.microsoft.com → Audit → New search
Parametros de busqueda recomendados
ActivitiesFileAccessed, FileAccessedExtended, ListItemsViewed
Userscuenta B2B de Circo o App Registration ID
Date rangePeriodo del engagement
B
Monitorear la App Registration en Entra ID
Entra ID → App registrations → CuraduriaM365-Circo-ReadOnly → Audit logs / Sign-ins
Desde esta pantalla se puede ver cada vez que la app se autentico, desde que IP, con que resultado y que usuario completo el MFA. Cada sesion del kit aparece como un evento de sign-in separado.
08 Revocacion al cierre del engagement Responsabilidad compartida

Al concluir el engagement, Circo Studio notifica formalmente al cliente para que revoque el acceso. La revocacion es responsabilidad del IT del cliente — Circo Studio no puede revocar sus propios accesos desde afuera.

01
Deshabilitar o eliminar la App Registration
Entra ID → App registrations → CuraduriaM365-Circo-ReadOnly → Delete
Eliminar la app invalida inmediatamente todos los tokens activos. Si se prefiere conservar el registro de auditoria, deshabilitarla en lugar de eliminarla: Overview → Enabled for users to sign-in → No.
02
Remover la cuenta B2B (Escenario B)
Entra ID → Users → [cuenta guest Circo] → Delete
Esto revoca el acceso de la cuenta guest a todos los sitios donde fue asignada como SCA. Verificar en el Unified Audit Log que no hubo actividad posterior a la fecha de notificacion de cierre.
03
Verificar en el Audit Log que no hay actividad posterior
Correr la busqueda en Purview (seccion 07) con el rango de fechas posterior a la notificacion de cierre. No deberia aparecer ninguna actividad de la app ni de la cuenta B2B.
09 Configurar el Client ID en el kit Ultimo paso

Con la App Registration creada y el Client ID copiado, pegarlo en el editor de configuracion del kit. Es el unico campo de acceso que requiere gestion manual.

01
Abrir el editor de configuracion
Abrir Config\kit_config_editor.html en el browser. No requiere servidor — funciona directamente desde el sistema de archivos.
02
Completar la seccion Identidad del cliente
Campos a completar en el editor
Nombre del clienteNombre comercial del cliente
Prefijo del tenantLa parte antes de .sharepoint.com
Client IDUUID copiado de Entra ID ← este paso
UPN cuenta B2BSolo Escenario B
Carpeta base localDonde se van a guardar los CSVs
03
Descargar el cliente_config.json
Usar el boton Descargar cliente_config.json en el editor. Guardar en Clientes\[NombreCliente]\Config\.
Este archivo es el que se pasa con -ConfigPath a todos los scripts del kit.
Verificar la conexion antes de ejecutar el relevamiento
# Test rapido de conexion con el Client ID configurado # Si la ventana de login se abre y completa sin error, la App Registration esta bien $config = Get-Content ".\Clientes\[Cliente]\Config\cliente_config.json" | ConvertFrom-Json $clientId = $config.cliente.client_id $tenantUrl= $config.cliente.sharepoint_url Connect-PnPOnline -Url $tenantUrl -ClientId $clientId -Interactive # Si la conexion es exitosa, probar una lectura basica Get-PnPSite Disconnect-PnPOnline Write-Host "Conexion OK - App Registration configurada correctamente" -ForegroundColor Green
